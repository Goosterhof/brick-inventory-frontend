# ADR-000: Why This Project Exists and How Decisions Are Made

**Date**: 2026-03-18
**Status**: accepted
**Transferability**: universal

This is not a technical decision. It is the foundation document for every decision that follows. Read this first.

---

## Who We Are

The owner of this repository is the co-owner and lead developer of **Script**, a company that teaches people programming. Graduates of Script join development teams — the owner leads a team of **20+ trainees and junior developers** building two production applications.

That team is the real audience for this project's decisions.

## Why This Project Exists

This LEGO Storage Inventory Management System is a real application, but its primary purpose is to serve as a **decision laboratory**. Ideas are tested here — in a smaller, controlled environment — before they are adopted by a team of 20+ people who will implement them literally and at scale.

The application is deliberately scoped to exercise the same architectural problems that appear in large production apps:

- Multi-app architecture with shared code
- Authentication and authorization
- CRUD operations with forms and validation
- API integration with type transformations
- Component libraries and design systems
- Testing and quality enforcement

Every technical decision made here is evaluated as if it will be adopted by a large team of juniors. **Simplicity, teachability, and clear rationale matter more than cleverness.**

## Why ADRs Are the Primary Deliverable

Junior developers and trainees lack the experience to push back on architectural decisions. If the lead says "we're doing it this way," the team does it that way — even if the reasoning is flawed.

This creates a dangerous feedback loop: bad decisions go unchallenged, get baked into codebases, and become "how we've always done it." By the time the consequences surface, the cost of changing course is enormous.

ADRs break that loop by forcing decisions through a structured process:

1. **State the problem honestly** — not the solution you already picked, but the actual forces in tension
2. **Evaluate real alternatives** — not strawmen set up to lose, but options you genuinely considered
3. **Record the reasoning** — so future readers understand _why_, not just _what_
4. **Acknowledge what you don't know** — open questions are not weakness; unacknowledged uncertainty is

The devil's advocate role in this project exists because the real team cannot fill it yet. Decisions need to survive genuine scrutiny here so they hold up when 20 people depend on them.

## How Decisions Should Be Evaluated

When evaluating any decision in this project, apply these lenses:

### 1. Would a junior understand this without asking?

If a pattern requires nuanced judgment to apply correctly, it is a bad pattern for this context. Rules should be mechanical: "always do X in situation Y" beats "use your best judgment."

### 2. What happens when someone follows this rule too literally?

Juniors will follow conventions to the letter. If strict adherence to a rule produces absurd results in edge cases, the rule needs a clearly documented escape hatch — not an unwritten understanding that "experienced developers know when to break it."

### 3. Does the decision scale to 50+ components and 10+ domains?

This project is small. The decisions should not be. Every choice is stress-tested against a codebase with 50+ components, 10+ domains, and 20 people interpreting the rules independently.

### 4. Can this be enforced automatically?

A rule that relies on code review to enforce is a rule that will be broken the week the lead is on vacation. Prefer decisions that can be enforced by linters, type checkers, CI pipelines, or project structure. If it can't be automated, document exactly how it should be reviewed.

### 5. Is the reasoning transferable?

Some decisions are specific to this LEGO app. Others apply to any large Vue/TypeScript project. ADRs should be honest about which category they fall into (see Transferability below).

## Transferability

Each ADR carries a **Transferability** label:

- **universal** — The reasoning applies to any large Vue/TypeScript project with a team of mixed experience levels. The LEGO app is just the vehicle; the decision would be made the same way in a different project.
- **project-specific** — The decision is tied to something particular about this application (the multi-app architecture, the LEGO design system, specific library choices). The _reasoning process_ may still be instructive, but the conclusion doesn't port directly.

This label is not a quality judgment. A project-specific decision is not worse than a universal one — it just has a different scope of applicability.

## The Roleplay Element

The CLAUDE.md file in this repository uses LEGO-themed metaphors and a corporate roleplay structure (Brick & Mortar Associates, CFO/CEO dynamic). This is intentional and serves a purpose:

- The owner spends all day surrounded by developers and developer jargon. This project combines the hobby of LEGO with development, and the roleplay creates mental distance from the day job while still doing real engineering work.
- When documentation is engaging to write and maintain, it actually gets written and maintained.
- The metaphors (supply warehouse, building inspectors, departments) map to real architectural concepts and make the codebase more navigable.

The roleplay does not reduce the rigor of decisions. The ADRs themselves are written straight — no metaphors, no characters. The fun lives in the process; the decisions are serious.

## War Room Governance

This firm operates as the **ADR development laboratory** for the war room (the cross-territory governance layer). Patterns are explored, proven, and battle-tested here before graduating to cross-project governance that affects ally territories.

What this means in practice:

- **Full ADR content, always** — This firm maintains complete ADR content in `decisions/` with sovereign numbering (001+). Other territories receive condensed "distilled projections" of cross-project ADRs; this firm gets the full version because the CEO needs to tweak, amend, and iterate freely.
- **Sovereign numbering** — Firm ADR numbers are independent from war room ADR numbers. Firm ADR-008 and war room ADR-0008 are unrelated decisions. When a firm pattern graduates to cross-project status, the war room assigns its own number.
- **Source end of the pipeline** — Changes proven here flow *up* to the war room for cross-project formalization. Projections flow *down* to other territories. This firm is upstream.
- **Cross-project ADRs that apply here** — War room ADR-0014 (Domain-Driven Frontend Structure). Canonical source for full ADR text: `adrs.script.nl`. Sovereign equivalent: ADR-014 (vertical slices) and ADR-008 (domain isolation enforcement).

This governance model is formalized in war room ADR-0015 (ADR Governance — Single Source of Truth, amended 2026-03-27).

## What This Document Is Not

This is not a project roadmap, a technical specification, or a style guide. Those live elsewhere. This document answers one question: **why should anyone trust the decisions recorded in this repository?**

The answer: because every decision was forced to justify itself against the needs of a real team, evaluated for the problems it would cause when followed literally at scale, and recorded with enough context that a future reader can judge whether the reasoning still holds.
