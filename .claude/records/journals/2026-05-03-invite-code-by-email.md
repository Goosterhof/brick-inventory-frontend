## Decisions Made

1. **Separate `useValidationErrors` instance for the email form** — Chose a dedicated `inviteEmailValidation` over reusing one of the existing field-error scopes. Each `useValidationErrors` call registers a response-error middleware that updates the same `errors` ref; sharing one across unrelated forms would smear field errors from one submit onto another's inputs. The composable is cheap, and the field type union (`'recipientEmail' | 'recipientName'`) makes the scope obvious to readers.
2. **`useFormSubmit` + try/catch combination, not pure try/catch** — `useFormSubmit` swallows 422 (so `useValidationErrors` can do its work), re-throws otherwise. To inline-render 429/500 without bubbling further, the action body catches non-422 errors itself and sets `inviteEmailError`. 422 explicitly re-throws so `handleSubmit` can swallow it. This preserves the established 422-via-middleware pattern while still surfacing 429/500 inline as the permit demands.
3. **New test file (`SettingsPageInviteEmail.spec.ts`) instead of extending `SettingsPageMembers.spec.ts`** — The members file already houses the existing invite-code generate/revoke/copy tests. Adding 10 more would push it past 40 tests in one file. The settings page already shards into Theme/Members/Config — one more shard for "invite by email" matches that rhythm and keeps each file focused.
4. **`[, tokenForm]` destructuring over `findAll('form').at(-1)`** — oxlint's type inference for `findAll<HTMLFormElement>` chained through `.at()` produced an `error` typed value (no-unsafe-call), and `forms[forms.length - 1]` triggered `unicorn(prefer-at)`. Array destructuring of two known forms is the only pattern that satisfies both lints without disabling either.
5. **Real `vue-router` in the RegisterPage integration test, mocked `vue-router` in the unit test** — The integration test exists to verify component composition through real components; mocking vue-router would defeat that. The unit test, which already mocks every other service, gets a hoisted `routeQuery` ref so each test can drive the query without a router instance.

## Self-Debrief

### What Went Well

- The middleware-awareness heuristic paid off immediately — a quick read of `services/http.ts` confirmed `deepSnakeKeys` middleware is registered, so the type stayed camelCase and the assertion in the test asserts camelCase too. No detour.
- Greping `/family/invite-code` upfront mapped the existing test blast-radius (members spec, theme spec, config spec) before I touched the page. I knew exactly which tests lived where.
- Splitting into `SettingsPageInviteEmail.spec.ts` instead of bolting onto the members file — the existing settings tests are already sharded by topic, and this fit the rhythm without forcing me to invent a new convention.
- The 422 re-throw + `useFormSubmit` swallow path landed in one go because I read `useFormSubmit.ts` and `useValidationErrors.ts` end-to-end before writing the action body.

### What Went Poorly

- **The form-index lint dance cost 4 cycles.** First `findAll('form').at(0)?.trigger('submit')` lit up `no-unsafe-call`. I switched to `forms[0]` — that triggered `prefer-at`. I switched to `forms.at(0)` (assigned to a variable) — `no-unsafe-call` came back. I switched to `forms[forms.length - 1]` for the rebrickable case — `prefer-at` again. Final winner: array destructuring. A pattern check for "how does this codebase handle multiple forms in one test wrapper" upfront would have saved 3 of those 4 attempts. There were no prior multi-form tests in this codebase.
- **Hit a `findTokenInput` race that needed an extra `flushPromises`.** When `members` resolved between `setValue` and `findAll('form')`, the email form rendered after I had already counted forms. Adding `await flushPromises()` immediately after mount fixed it. Should have done that first instead of debugging through "why is mockPutRequest call count 0."
- **The integration test broke before I noticed `useRoute()` needs a router.** I caught it on the first integration run, fixed in one shot, but I should have predicted: any new `useRoute()` consumer needs a router for `mount()`. That's a one-line check before running.

### Blind Spots

- I almost shipped without verifying the existing `findComponent(TextInput)` (singular) calls in `SettingsPageConfig.spec.ts`. Coverage was passing because the assertion-failing tests still executed every line — the failure was on a `props('error')` value mismatch, not a coverage gap. The just-graduated heuristic ("grep ALL test files for `findComponent(ComponentName)` singular calls") caught it on the first test run, but I should have done the grep before changing the page rather than after.
- I didn't check `route.query.invite` against any URL-encoding edge case (`?invite=BRICK%20WHITESPACE`). The permit's "string-or-array-or-undefined" framing is satisfied, but I notice I've assumed Vue Router has already URL-decoded by the time we read `query.invite`. It does — that's documented behaviour — but I didn't verify by reading the router source.
