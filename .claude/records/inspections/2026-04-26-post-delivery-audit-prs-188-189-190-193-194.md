## Findings

### Architecture & API Surface

1. **Fixed footprint breaks sibling component API contract** `medium`
    - **Location:** `src/shared/components/LegoBrickCuboidCss.vue`, `src/shared/components/LegoBrickIsometricSvg.vue`
    - **Standard:** Component API consistency (CLAUDE.md — Every brick has a purpose, every connection is deliberate)
    - **Observation:** All other shape components in the family (`LegoBrick`, `LegoBrickSvg`, `LegoBrickSideSvg`, and all other `Lego*` variants) expose `columns` and `rows` props with sensible defaults. The two new 3D components hard-code a 6×2 footprint with no configurable dimensions. A senior reviewing the component catalogue sees 14 shapes that support arbitrary footprints and 2 that silently don't. The JSDoc says "6×2, fixed footprint" but not _why_ the constraint exists (e.g., 3D perspective calibrated for exactly 6×2). If this is deliberate, it needs a WHY comment. If it's an omission, `columns`/`rows` props should be added.
    - **Recommendation:** Add a WHY comment on the component explaining the fixed-footprint constraint, or add `columns`/`rows` props to align with the family API. No silent divergence from the established prop API.

2. **`prevCursor` is a dead type field** `medium`
    - **Location:** `src/apps/families/types/part.ts:74`
    - **Standard:** No dead code (CLAUDE.md — no dead bricks; knip policy)
    - **Observation:** `CursorPaginatedParts.prevCursor` is declared in the type and appears in both the unit test fixture (`PartsPage.spec.ts:126`) and the integration test envelope helper (as `prev_cursor` in the pre-conversion fixture). However, no production page reads this field — `PartsPage.vue` uses only `nextCursor` and pagination is forward-only. The field represents a real API response value, so it's not dead in the API sense, but it is dead in the app sense. No comment explains whether backward pagination is planned, abandoned, or simply not needed.
    - **Recommendation:** Add a WHY comment: `// API returns prevCursor but pagination is forward-only; retained for type accuracy.` _(Fixed in this PR.)_

### Tests

3. **Download tests use `as unknown as HTMLAnchorElement` casts — ADR-012 violation** `low`
    - **Location:** `src/tests/unit/shared/helpers/bricklinkWantedList.spec.ts`
    - **Standard:** ADR-012 — no `as unknown` type assertions in tests; use typed factory helpers
    - **Observation:** Both download tests (`downloadBrickLinkWantedList`) create a mock anchor element via `{...} as unknown as HTMLAnchorElement`. ADR-012's typed pattern is fully applicable here — this is a plain TypeScript helper test, not a Vue component test. The `as unknown` double cast silences TypeScript on an object missing ~60 `HTMLAnchorElement` properties. The behaviour tested is meaningful, but the cast is the exact anti-pattern ADR-012 was written to eliminate.
    - **Recommendation:** Replace `as unknown as HTMLAnchorElement` casts with a typed partial helper (e.g., `createMockAnchor()` returning a spy-equipped partial). This is the DOM-test equivalent of the typed prop factories already used in component tests.

### Documentation

4. **Parts domain API section missing `GET /family/parts`** `low`
    - **Location:** `.claude/docs/domain-map.md` — parts domain entry
    - **Standard:** Documentation accuracy (CLAUDE.md — paper trail, no gaps)
    - **Observation:** The parts domain API entry documents only `GET /family-sets/missing-parts` (the PartsMissingPage endpoint added by PR #189). `PartsPage.vue` calls `GET /family/parts` with cursor pagination — this endpoint was present before PR #189 and is entirely absent from the domain map. Third consecutive occurrence of the domain-map-underdocumented pattern (prior: `brick-dna` 2026-03-29; `BrickShapes`/`PageTransitionDemo` 2026-04-11; `PartsMissingPage` route 2026-04-20 — each previously flagged). Three occurrences confirm this is a structural process gap.
    - **Recommendation:** Add `GET /family/parts` to the parts domain API entry. _(Fixed in this PR.)_ Structural fix: add a domain map update step to the permit/journal template so architects verify the map on every delivery.

5. **`<style scoped>` in `LegoBrickCuboidCss.vue` is ADR-003 compliant** `info`
    - **Location:** `src/shared/components/LegoBrickCuboidCss.vue`
    - **Standard:** ADR-003 — UnoCSS-first; `<style scoped>` permitted for CSS properties not expressible as attributify
    - **Observation:** `transform-style: preserve-3d`, `perspective`, `perspective-origin`, and `transition` on hover states are genuinely not expressible as UnoCSS attributify. ADR-003 explicitly permits `<style scoped>` as an escape hatch for these cases. No violation. Flagging as info — this was verified, not missed.

6. **No retry mechanism on fetch failure in `PartsMissingPage`** `info`
    - **Location:** `src/apps/families/domains/parts/pages/PartsMissingPage.vue`
    - **Standard:** UX consistency (other pages e.g., `PartsPage` show an error banner but also offer retry)
    - **Observation:** On fetch failure, `PartsMissingPage` displays an error banner and sets `loadError = true` but offers no "try again" button. The user must navigate away and back to retry. For a shopping list page that depends on network availability, this is a UX gap. Intentional or not — not a code defect, but worth a CEO decision.
