## Findings

### Architecture

1. **`prevCursor` field declared but never consumed** `severity: low`
    - **Location:** `src/apps/families/types/part.ts:73` — `CursorPaginatedParts.prevCursor`
    - **Standard:** ADR-004 — in-app types should represent what the code actually uses
    - **Observation:** `prevCursor: string | null` was added as part of the camelCase rename from `prev_cursor`. The field exists in the API response and was in the original type. However, it is not read anywhere in `PartsPage.vue` — cursor pagination is forward-only (load-more), and `PartsMissingPage` uses a different API with no cursor at all. The field is dead type surface today. `knip` may not flag it since it's an interface member, not a standalone export.
    - **Recommendation:** If backward pagination is not planned, remove `prevCursor` from the interface to keep the type honest. If it's intentionally retained for future use, add a comment documenting that intent (this is one of the rare cases where a WHY comment is warranted — the field looks like a typo omission otherwise).

### Tests

2. **Sort chip test uses optional chaining without prior assertion — vacuous pass risk** `severity: medium`
    - **Location:** `src/tests/integration/apps/families/domains/parts/pages/PartsMissingPage.spec.ts` — `it("reorders PartListItems when a sort FilterChip is clicked")`
    - **Standard:** Firm test reliability standards — a test that passes when the thing under test does nothing is worse than no test
    - **Observation:** `await nameChip?.find("button").trigger("click")` uses optional chaining. If `nameChip` is `undefined` (e.g., chip text changed, chip not rendered), the entire expression short-circuits to `undefined`. `flushPromises()` is awaited next, but nothing was clicked. The subsequent assertion `expect(namesByName).toEqual(["Axle 2", "Brick 2x4"])` then compares against the unmodified shortfall-ordered list `["Brick 2x4", "Axle 2"]` — which would **fail**, catching the bug. So the test is NOT actually vacuous in this case: a missing chip will produce a list in shortfall order that differs from the expected name order, and the test will fail. However, the reliance on the sort order differing to catch chip-not-found is fragile. If the expected values were coincidentally sorted the same way, the test would pass silently.
    - **Recommendation:** Replace the optional chain with a direct assertion first: `expect(nameChip).toBeDefined()` before `nameChip!.find("button").trigger("click")`. This makes intent explicit and eliminates all vacuous-pass scenarios regardless of the data fixtures.

### Docs

3. **Overlap with PR #192 creates merge conflict risk** `severity: medium`
    - **Location:** `.claude/docs/domain-map.md` — parts domain section
    - **Standard:** Operations Protocol — keep the paper trail clean, avoid redundant delivery
    - **Observation:** PR #192 (`claude/charming-newton-EB5XJ`, commit `ddbf44c`) and PR #193 (`claude/charming-newton-7CEcm`, commit `df42e02`) both modify `domain-map.md` for the same parts domain entry. PR #192 changes: adds `/parts/missing` to routes, adds `PartsMissingPage` to pages, adds `GET /family-sets/missing-parts` to API. PR #193 changes: same routes/pages plus also adds `GET /family/parts` to API. PR #193 is the strict superset. If PR #192 is merged first, PR #193 will have a merge conflict in `domain-map.md`. If PR #193 is merged first, PR #192 becomes fully redundant. Neither PR references the other.
    - **Recommendation:** Close PR #192 as superseded by PR #193. Add a note in the PR #192 close comment referencing PR #193.

### Generated Code

4. **`component-registry.json` churn update for `LegoBrickIsometricSvg` / `LegoBrickSideSvg` appears unrelated to this fix** `severity: low`
    - **Location:** `src/shared/generated/component-registry.json`
    - **Standard:** Commit hygiene — commits should contain what their message says they contain
    - **Observation:** The commit `df42e02` updates `churn.commits` and `churn.linesChanged` for `LegoBrickIsometricSvg` (0 → 1, 0 → 236) and `LegoBrickSideSvg` (0 → 1, 0 → 254). These are SVG components added in the earlier 3D bricks feature (PR #188), not touched by this fix. The registry appears to be auto-generated and was presumably updated as a side effect of a pre-commit or build hook. The churn numbers reflect the PR #188 delivery, not this commit.
    - **Observation 2:** If the registry is auto-generated, it should be regenerated consistently on every commit so it stays current. The fact that it's updating _now_ (in this unrelated commit) suggests the registry generation hook ran here but did not run at the PR #188 commit time — meaning the registry was stale between PR #188 and this commit.
    - **Recommendation:** Verify the component registry generation hook fires consistently. A stale registry between features undermines the Showcase's accuracy. If it's not firing reliably, treat this as a tooling gap.
