## Findings

### Documentation

**1. `brick-catalog.md` does not exist** `medium`

- **Location:** `.claude/docs/` — file is absent. Referenced from `domain-map.md` (line 92) and by `npm run lint:catalog` (which crashes with ENOENT).
- **Standard:** SOP 3 — documentation that doesn't match reality is worse than no documentation; CLAUDE.md "The building we're constructing" section references the Brick Catalog as the canonical parts inventory.
- **Observation:** `domain-map.md` directs readers to "See the [Brick Catalog](./brick-catalog.md) for the full parts inventory with props, slots, emits, and composition patterns." The file does not exist. Additionally, `npm run lint:catalog` crashes with ENOENT because the script (`scripts/validate-brick-catalog.mjs`) attempts to open the file. This is a documented, runnable npm script that is silently broken. 31 shared components have no canonical props/slots/emits documentation. A developer or client reviewer following the domain-map link hits a dead end.
- **Recommendation:** Either create `brick-catalog.md` documenting all 31 components (the correct answer — this is the portfolio's component reference), or remove the link from `domain-map.md` and remove `lint:catalog` from `package.json` to eliminate the broken script. Creating the catalog is strongly preferred: a portfolio piece without documented component contracts is incomplete.

---

**2. `component-registry.json` persistently fails `format:check`** `low`

- **Location:** `src/shared/generated/component-registry.json`
- **Standard:** CLAUDE.md formatting standards; format:check is part of the pre-push gauntlet.
- **Observation:** This file has failed format:check on both the 2026-03-24 inspection and this inspection. The file is auto-generated by a pre-commit hook. The generator produces output that does not match oxfmt's expected formatting, so every generation re-introduces the violation. Content is accurate (31 components, correct entries). Format only.
- **Recommendation:** Fix by running `npm run format`. Then investigate whether the registry generation script should pipe its output through `oxfmt` before writing, or whether `npm run format` should be added after the generation step in the pre-commit hook sequence. Without a fix to the generation pipeline, this will reappear on every commit that triggers registry regeneration.

- **Location:** `src/shared/services/router/components.ts`, line 21 — `// TODO :: use the proper 404 component`
- **Standard:** SOP 5 — assess age and severity of TODO/FIXME comments.

### Patterns

**4. `lint:catalog` is a broken npm script** `medium`

- **Location:** `package.json`, `scripts/validate-brick-catalog.mjs`
- **Standard:** Every documented npm script must be executable (implied by CLAUDE.md's Inspection Checklist philosophy — if it's listed, it must work).
- **Observation:** `npm run lint:catalog` crashes immediately with ENOENT because `brick-catalog.md` does not exist. The script is listed in `package.json` and therefore communicates to any reader that catalog validation is a supported operation. That it silently fails on first invocation is misleading. This is directly linked to Finding 1 (missing brick-catalog.md) — both should be resolved together.
- **Note:** This script is not in the pre-push gauntlet, so it does not block development. However, broken scripts in a portfolio repo undermine the impression of craft.
- **Recommendation:** Resolve together with Finding 1. If brick-catalog.md is created, verify the script runs cleanly. If the catalog is removed from the roadmap, remove the script from package.json.

### Code Quality Observations (Non-Findings)

The following were investigated and confirmed to be deliberate, not violations:

- **`console.error` in `storage.ts`** — 5 calls. `.oxlintrc.json` explicitly overrides `no-console` and `no-restricted-globals` for `src/shared/services/storage.ts`. The storage service IS the abstraction layer for localStorage; surfacing errors at this boundary is intentional. Not a violation.
- **`@/apps/` import paths in test files** — Technically forbidden by CLAUDE.md import conventions, but `.oxlintrc.json` overrides `no-restricted-imports: "off"` for all `src/tests/**` files. Test files operate under different import rules than production code. Deliberate config decision, not a violation.
- **`eslint-disable` comments in `components.ts`** — `/* eslint-disable vue/one-component-per-file */` and two `// eslint-disable-next-line vue/require-prop-types` comments. Oxlint supports eslint-disable syntax. The vue/one-component-per-file suppression is warranted (two factory functions each define a component; splitting to separate files would destroy the cohesion of the factory pattern). The vue/require-prop-types suppression follows Vue's own documented pattern for functional components with runtime props declaration. Not violations.
- **`guards.ts` returns booleans, not `next()`** — Auth guard returns boolean values, not the deprecated `next()` callback. The Vue Router deprecation warnings seen during test execution originate from Vue Router's internal API, not from this codebase's guard implementation. Code is correct per Vue Router 5's current API.
- **No raw `useRouter`/`useRoute`/`RouterLink`** — Confirmed absent across all shared code. ADR-001 is respected. The `createRouterLink` factory in `components.ts` renders an `<a>` element with click handler — it IS the RouterService's own link component, not a raw Vue Router consumer.
- **No `istanbul ignore` or `v8 ignore` comments** — Confirmed absent. ADR-005 is respected throughout shared.
- **Service factories, not singletons** — All 7 shared services export `create*()` functions. ADR-002 is respected.
- **`Object.freeze()` in resource-adapter.ts** — Confirmed present. ADR-006 immutability pattern is correctly implemented.
