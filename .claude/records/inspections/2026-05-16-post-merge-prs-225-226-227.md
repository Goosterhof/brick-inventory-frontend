## Findings

### Category: Tests / Correctness

1. **Incomplete fix shipped in #226, required same-day follow-up in #227** `severity: medium`
    - **Location:** `src/apps/families/domains/sets/pages/SetsOverviewPage.vue:68`
    - **Standard:** Ship-complete fixes; tests cover the full input space before merge
    - **Observation:** PR #226 changed the filter from `s.setNum.toLowerCase()` to `(s.set?.setNum ?? '').toLowerCase()`. This fixed the TypeError for normal sets (where `s.set?.setNum` is populated) but silently broke wishlist sets ‚Äî which carry no nested `set` ‚Äî by falling back to `''`, making them invisible to any search. PR #227 corrected this to `?? s.setNum` the next morning. The root cause: #226 was written from the perspective of the normal-set code path only; the wishlist code path was not considered until a test was written later.
    - **Recommendation:** Before merging a fix to a filtered list, enumerate all item subtypes (owned, wishlisted, sealed, etc.) and add at least one test for each subtype's field access pattern. The existing `status: 'wishlist' as const` mock in the spec suite provides a ready-made fixture.

2. **Missing terminal null guard on `setNum` in search filter** `severity: low`
    - **Location:** `src/apps/families/domains/sets/pages/SetsOverviewPage.vue:68`
    - **Standard:** Defensive null access; paired lines should use the same guard pattern
    - **Observation:** Current code: `const setNum = (s.set?.setNum ?? s.setNum).toLowerCase()`. The adjacent `name` line uses a terminal `?? ''` guard: `const name = (s.set?.name ?? '').toLowerCase()`. If `s.setNum` is ever `undefined` (e.g., a newly-constructed domain object before persistence), `.toLowerCase()` will throw. In practice `setNum` is the primary identifier and should always be populated, but the asymmetry with the `name` line is a latent inconsistency.
    - **Recommendation:** Change to `(s.set?.setNum ?? s.setNum ?? '').toLowerCase()` for parity with the `name` guard. One-character fix, eliminates the asymmetry.
